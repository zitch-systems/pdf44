## Accessibility Audit (WCAG 2.2 AA)

### Prior state: 41/100

The app had solid foundations in places (skip link, landmark roles, modal focus trap + Escape, `lang` attribute, `prefers-reduced-motion` block, radio/checkbox labels wrapped, `aria-hidden` on decorative hub icons, `aria-expanded` on mobile toggle) but failed on several WCAG A/AA criteria that block keyboard and screen-reader users entirely.

### Fixed this round

**A11Y-1 CRITICAL — Sidebar navigation was entirely keyboard-inaccessible**
- Location: `index.html:2692–2747` (`renderSidebar`)
- Root cause: Every nav item was `<div onclick="route(...)">` — not in tab order, no role, no keydown handler.
- Fix: Converted all nav items to `<a href="/{slug}">` with `onclick="route(...);return false;"` and `aria-current="page"` on the active item. Uses real hrefs for middle-click/right-click/prerender, JS router for SPA navigation.

**A11Y-2 CRITICAL — No global `:focus-visible` indicator**
- Location: `index.html:~1351` (new CSS block)
- Root cause: Only `.form-input:focus` had a styled ring; all buttons, links, nav items, tool cards, the range slider lacked any visible focus indicator (WCAG 2.4.7).
- Fix: Added `:where(a, button, [role="button"], input, select, textarea, .tool-card, .quick-tool-btn, .modal-close, .theme-toggle, .nav-item, summary):focus-visible` with `outline: 2px solid var(--accent); outline-offset: 2px; box-shadow: 0 0 0 4px rgba(229,50,45,0.25)`. Added dedicated `input[type="range"]:focus-visible` ring (the old code removed `outline` with no replacement).

**A11Y-3 HIGH — Form labels not programmatically associated with inputs**
- Location: `index.html:7063–7400` (`getToolSettingsHTML` switch)
- Root cause: `<label class="form-label">Text</label>` siblings (not wrappers) of `<input id="x">` with no `for` attribute — screen readers announce controls without names.
- Fix: Added `for="inputId"` to every un-associated `.form-label` across all tool settings (split, rotate, remove, extract, compress, pdf2jpg/png, pdf2word, pdf2ppt, jpg2pdf, html2pdf, text2pdf, ocr, edit, watermark, pagenumber, crop, protect, unlock, sign, redact, background, bates, header, formfill, merge).

**A11Y-4 HIGH — Toast container had no `aria-live` region**
- Location: `index.html:2318` (markup), `9066–9077` (`toast()`)
- Root cause: `#toasts` injected nodes with no live-region semantics — all success/error feedback was silent to screen readers (WCAG 4.1.3).
- Fix: Added `role="status" aria-live="polite" aria-atomic="false"` to the container. In `toast()`, error and warning nodes get `role="alert"` so they interrupt immediately; others inherit the polite container. Toast icon SVGs get `aria-hidden="true"`.

**A11Y-5 HIGH — Generic modal lacked dialog semantics and didn't return focus**
- Location: `index.html:2307–2316` (markup), `9078–9109` (`openModal/closeModal`)
- Root cause: `#modal` had no `role="dialog"`, no `aria-modal`, no `aria-labelledby`, no `aria-hidden` in initial state. Close button was `×` with no `aria-label`. Focus was not returned to the trigger on close.
- Fix: Added `role="dialog" aria-modal="true" aria-labelledby="modalTitle" aria-hidden="true"` to the overlay. Added `aria-label="Close dialog"` to the close button. `openModal()` now saves `document.activeElement` as `overlay._trigger`; `closeModal()` restores focus to it.

**A11Y-6 HIGH — SPA route changes didn't manage focus or announce the new page**
- Location: `index.html:3618` (end of `route()`)
- Root cause: SPA navigation swapped content without moving focus — keyboard users were left at the previous position with no indication of the page change (WCAG 2.4.3, 3.2.3).
- Fix: After `render()`, `route()` now calls `requestAnimationFrame(() => { main.setAttribute('tabindex', '-1'); main.focus(); })` to move focus to `#main-content`, giving screen readers the new page context.

**A11Y-12 MEDIUM — `prefers-reduced-motion` block too narrow**
- Location: `index.html:1351–1353`
- Root cause: Only 4 specific animation selectors were in the block; `@keyframes pulse`, `slideUp`, `fadeIn`, `scaleUp`, `spin` etc. still ran.
- Fix: Added `*, *::before, *::after { animation-duration: 0.01ms !important; animation-iteration-count: 1 !important; transition-duration: 0.01ms !important; }` as the standard blanket override (WCAG 2.3.3 / 2.3.1).

**POPSTATE-1 LOW — Duplicate `popstate` listener double-fired `route()`**
- Location: `index.html:3632–3634` (first listener), `10360–10363` (second)
- Root cause: Two separate `window.addEventListener('popstate')` calls both invoked `route()`, causing every Back/Forward to route twice and re-render.
- Fix: Removed the first listener (simpler, only used `detectToolFromLocation`); the second listener at ~10360 handles both `e.state.tool` and `detectToolFromLocation` as fallback.

## Performance Audit

### Score: 48/100 (structural debt; quick-wins applied)

**PERF-1 CRITICAL — 573KB inline script is render-blocking on every route**
- The single inline `<script>` block (~573KB) runs synchronously on every page load before any content renders. LCP and FCP are gated on parsing this entire bundle.
- *Mitigation applied:* No quick fix without a fundamental build-system change. Documented for future roadmap: split app into a `<script defer src="app.js">` loaded separately, allowing initial HTML to paint first.
- *Note:* The static prerender (`build.js`) does embed a visible `<h1>` and `<p>` in `#content` before JS runs, which partially mitigates LCP for crawlers. For real users, the inline script still blocks.

**PERF-2 HIGH — 212 × 690KB prerendered copies = ~142MB dist**
- Every tool page is a full copy of `index.html` with only head metadata swapped. Same 573KB script, same 48KB CSS, duplicated 212 times.
- *Roadmap:* Extract JS/CSS to separate versioned files; prerendered pages become lightweight shells.

**PERF-3 HIGH — LCP hero renders into empty `#content` via JS**
- The initial `<div id="content"></div>` is empty; `renderHome()` fills it. Prerender injects a static stub (`build.js:163–171`) but JS immediately replaces it, causing a potential FCP/LCP flash.
- *Partial fix:* The prerendered stub text is semantically identical to the JS output, so crawlers and no-JS users see real content. LCP for logged-in returning users should be checked via RUM.

**PERF-6 LOW — CF Analytics beacon loads with unconfigured placeholder token**
- Location: `index.html:88–89`
- The `data-cf-beacon='{"token": "REPLACE_WITH_CF_ANALYTICS_TOKEN"}'` tag loads the beacon JS even with a placeholder, wasting a network round-trip on every page.
- *Recommendation:* Either configure the real token or remove the `<script>` tag until configured.

**PERF-10 LOW — DPR canvas cap comment says 2 but code varies**
- In `processCompress`, the DPR used is the configured DPI/72 (not device DPR), which is correct. The README mention of "capped at 2" refers to thumbnail rendering. No code bug confirmed.

## Code Quality / Functional Regressions

**MUT-1 / ARCH-1 HIGH — `processRotate` mutated the shared cached `pdfDoc`**
- Location: `index.html:8054–8073`
- Root cause: `processRotate` called `f.pdfDoc.getPage(i).setRotation(...)` directly on the cached document. Re-clicking "Process" stacked rotations (90° twice = 180°); running a second tool after Rotate saw a pre-rotated document.
- Fix: `processRotate` now copies all pages into a fresh `PDFDocument`, applies rotation to the copy, and saves — leaving `f.pdfDoc` untouched.

**SPLIT-1 MEDIUM — `processSplit` single-range path had no bounds check**
- Location: `index.html:8013–8020`
- Root cause: The multi-range path (line 8026) had `if (s < 0 || e >= pc || s > e) continue;` but the single-range fast-path had none — out-of-range input crashed with an uncaught `pdf-lib` error.
- Fix: Added the same bounds check before the single-range copy, with a user-facing error toast.

**POPSTATE-1 LOW — Duplicate `popstate` listener (see A11Y-6 above)**
